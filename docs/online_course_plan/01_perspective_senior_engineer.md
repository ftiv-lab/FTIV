# 視点1: スーパーシニアエンジニア (技術・品質・設計)

**テーマ案: "AI時代の「本物のエンジニアリング」入門"**

この視点では、単に「アプリが動けばいい」というレベルを超えて、AIを使っていても（あるいは使っているからこそ）疎かになりがちな「品質」「保守性」「安全装置」に焦点を当てます。

## コンセプト
*   **「AIは優秀な部下、責任はあなた」**: AIにコードを書かせても、品質を担保するのは人間の役割。
*   **「High Gravity Quality」**: 徹底した型安全性、テスト、Lint、Git運用。
*   **「失敗から学ぶ」**: マインドマップ機能の追加と削除（ピボット）の実例を通して、技術的負債と意思決定を学ぶ。

## カリキュラム提案 (技術特化)

### Module 1: AI開発のための「要塞」を築く (Infrastructure)
*   **仮想環境の鉄則**: `venv` と `requirements.txt` の完全管理。バージョン固定の重要性。
*   **品質ゲート**: `ruff` (Linter) と `pytest` を「呼吸するように」実行する環境作り。
*   **ワンクリック検証**: `verify_all.bat` の作成。CI/CDの第一歩はローカルスクリプトから。

### Module 2: AIを制御する「ルール」 (Governance)
*   **システムプロンプト設計**: `.agent/RULES.md` や `rules.yml` でAIに「性格」と「制約」を与える。
*   **ドキュメント駆動開発 (DDD)**: いきなりコードを書かせない。`implementation_plan.md` で設計し、レビュしてから実装させるフロー。
*   **タスク管理**: `task.md` で進捗を可視化し、AIの迷走を防ぐ。

### Module 3: 堅牢な設計パターン (Architecture)
*   **疎結合 (Decoupling)**: Signal/Slot を使った「親を知らない子」コンポーネントの設計。
*   **DI (Dependency Injection)**: AIがテストしやすいコードを書くための依存性注入。
*   **混沌への備え**: `ConfigGuardian` のような自己修復機能の実装。

### Module 4: 実践ケーススタディ (The Pivot)
*   機能追加の誘惑: マインドマップモードの実装。
*   複雑性の罠: コードがスパゲッティ化する予兆を検知する。
*   **Gitによる外科手術**: `feature` ブランチへの退避と、`v1.0` リリースのための断捨離決断。

## 差別化ポイント
*   「作って終わり」の初心者講座ではない。「運用し続ける」「拡張し続ける」ためのプロの作法を教える。
*   FTIVの実コードを教材にし、「なぜここでこのリファクタリングが必要だったか」を生々しく解説できる。
